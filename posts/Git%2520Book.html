Git 基础入门学习笔记 ​

Git 是分布式版本控制系统。它记录文件的变化，让开发者能够查看历史、协作开发、创建分支，并在需要时恢复到可靠版本。

1. Git 的核心概念 ​

Git 的日常操作围绕三个区域展开：

区域含义常用命令
工作区（Working Tree）正在编辑的项目文件编辑器、git status、git diff
暂存区（Staging Area）准备进入下一次提交的改动git add、git restore --staged
本地仓库（Repository）已提交的版本历史git commit、git log
远程仓库（Remote）GitHub、GitLab 等托管的仓库git fetch、git pull、git push
text
工作区 -- git add --> 暂存区 -- git commit --> 本地仓库 -- git push --> 远程仓库

Git 跟踪的是提交（commit）形成的历史图，而不是简单地保存每个文件的副本。

2. 安装与初始配置 ​

检查 Git 是否安装：

bash
git --version

首次使用应配置用户名和邮箱，它们会写入之后的提交记录：

bash
git config --global user.name "你的名字"
git config --global user.email "your-email@example.com"

推荐的基础配置：

bash
# 将默认初始分支设为 main
git config --global init.defaultBranch main

# 提交时自动转换 Windows / Linux 换行符
git config --global core.autocrlf true      # Windows
# git config --global core.autocrlf input   # macOS / Linux

# 查看所有全局配置
git config --global --list

3. 创建仓库与克隆项目 ​

3.1 初始化本地仓库 ​

bash
mkdir my-project
cd my-project
git init
git status

初始化后，项目目录中会生成隐藏的 .git 目录。它保存 Git 历史、配置和对象数据，不能随意删除。

3.2 克隆远程仓库 ​

bash
# HTTPS 地址
git clone https://github.com/用户名/仓库名.git

# SSH 地址
git clone git@github.com:用户名/仓库名.git

# 克隆到指定目录
git clone https://github.com/用户名/仓库名.git my-project

# 只克隆指定分支
git clone --branch develop --single-branch https://github.com/用户名/仓库名.git

4. 日常提交工作流 ​

最常见的工作流是：查看状态 → 查看差异 → 暂存改动 → 提交。

bash
git status
git diff
git add README.md
git add src/
git commit -m "feat: add user login"

暂存所有已跟踪文件的修改和删除：

bash
git add -u

暂存当前目录下所有改动，包括新文件：

bash
git add .

查看暂存区内容：

bash
git diff --staged

推荐的提交信息格式：

text
<type>: <简短说明>

例如：
feat: add user login
fix: handle empty avatar
docs: update deployment guide
refactor: simplify order service

常用 type：

类型说明
feat新功能
fix修复缺陷
docs文档变更
style格式调整，不改变逻辑
refactor重构，不改变功能
test测试变更
chore构建、依赖、工具等杂项

5. 查看状态与历史 ​

bash
git status                 # 当前分支和改动状态
git diff                   # 工作区与暂存区的差异
git diff --staged          # 暂存区与最近提交的差异
git log                    # 完整提交历史
git log --oneline          # 一行显示一条提交
git log --graph --oneline --all  # 图形化查看分支历史
git show HEAD              # 查看最近一次提交详情
git show <提交ID>          # 查看指定提交
git reflog                 # 查看 HEAD 的移动记录，可用于找回误操作

6. 忽略不需要提交的文件 ​

在仓库根目录创建 .gitignore：

gitignore
# 依赖目录
node_modules/

# 构建产物
dist/
target/

# 环境变量与本地配置
.env
.env.local

# IDE 文件
.idea/
.vscode/

# 日志
*.log

规则说明：

  • *.log：忽略所有 .log 文件。
  • /dist：仅忽略根目录的 dist。
  • logs/：忽略任意位置名为 logs 的目录。
  • !important.log：不忽略指定文件。

已被 Git 跟踪的文件不会因为后来加入 .gitignore 而自动取消跟踪：

bash
git rm --cached 文件名
git commit -m "chore: stop tracking local config"

7. 分支管理 ​

分支用于隔离不同的开发工作。不要直接在团队共享的 main 分支上开发功能。

bash
git branch                       # 查看本地分支
git branch -a                    # 查看本地和远程分支
git switch -c feature/login      # 新建并切换分支（推荐）
git switch develop               # 切换到已有分支（新版本写法）
git checkout feature             # 切换到已有分支（旧版本写法）   
git branch -m old-name new-name  # 重命名当前或指定分支
git branch -d feature/login      # 删除已合并分支
git branch -D feature/login      # 强制删除未合并分支

典型功能开发流程：

bash
#切主干 → 2. 更新主干到远程最新版本 → 3. 基于最新主干新建登录功能分支
git switch main
git pull --ff-only origin main
git switch -c feature/login

# 编码、测试、提交
git add .
git commit -m "feat: add login page"

git push -u origin feature/login

-u 会建立本地分支与远程分支的跟踪关系；之后在该分支上可直接执行 git push 和 git pull。

8. 合并与变基 ​

8.1 merge：保留分支历史 ​

bash
git switch develop
git merge feature/login

若想始终创建一个合并提交：

bash
git merge --no-ff feature/login

8.2 rebase：整理线性历史 ​

  • rebase作用：把当前分支所有提交 “剪下来”，贴到目标分支最新节点后面，提交历史变成一条直线，不会产生多余 merge commit，代码历史干净。
  • git cherry-pick：单独摘取某一条提交，复制到当前分支
    作用：只拿某一个 commit，不拿整条分支，适合只需要别人分支里某一处修改，不用合并全部分支。
bash
#切换到你本地的登录功能开发分支
git switch feature/login
#拉取远程仓库 origin 所有分支最新提交记录，但不合并、不改动本地代码
git fetch origin
#把你 feature/login 上所有自己的提交，剪切下来，重新放到远程最新的 main 代码后面
git rebase origin/main

冲突解决后：

bash
git add 冲突已解决的文件
git rebase --continue

放弃本次变基：

bash
git rebase --abort

不要对已经推送、且可能被他人基于其开发的公共分支随意执行 rebase。它会重写提交历史，可能造成协作混乱。

9. 远程仓库同步 ​

查看远程仓库：

bash
git remote -v
git remote show origin

添加、修改和删除远程仓库：

bash
git remote add origin git@github.com:用户名/仓库名.git
git remote set-url origin git@github.com:用户名/新仓库名.git
git remote remove origin

同步远程数据：

bash
git fetch origin                # 一次性拉取远程所有分支（main、feature/api、hotfix…）的最新记录
git pull --ff-only origin main  # 下载并只允许快进合并
git push origin main            # 推送 main 分支
git push -u origin feature/api  # 推送并关联新分支

git pull 可以理解为 git fetch 加上合并或变基。需要先检查远程变化时，优先使用 git fetch。

删除远程分支：

bash
git push origin --delete feature/old-page

清理已删除的远程分支引用：

bash
git fetch --prune

10. 撤销与恢复 ​

Git 的撤销操作必须先分清：改动是否已经暂存、是否已经提交、是否已经推送。

10.1 放弃工作区修改 ​

bash
git restore 文件名
git restore .

这会丢弃未暂存的本地修改，不能轻易恢复。

10.2 取消暂存 ​

bash
git restore --staged 文件名

文件改动仍留在工作区，只是从暂存区移除。

10.3 修改最近一次提交 ​

bash
git commit --amend -m "feat: correct commit message"

也可先 git add 漏掉的文件，再执行 git commit --amend。如果该提交已推送到公共分支，不应随意使用 amend。

10.4 安全地撤销已提交内容 ​

bash
git revert <提交ID>
git revert HEAD

git revert 会创建一个新的反向提交，适合已经推送到公共分支的提交。

10.5 reset：移动分支指针 ​

bash
git reset --soft HEAD~1   # 撤销提交，保留暂存区和工作区
git reset --mixed HEAD~1  # 撤销提交，保留工作区（默认）
git reset --hard HEAD~1   # 丢弃提交、暂存区和工作区修改

git reset --hard 会丢弃本地未保存的改动。操作前先用 git status、git diff 确认，必要时先创建临时分支或 stash。

10.6 找回误删提交 ​

bash
git reflog
#新建分支 recover-work 并自动切过去
git switch -c recover-work <提交ID>

11. 暂存现场：git stash ​

当工作尚未完成但需要切换分支时，可以临时保存当前改动：

bash
git stash push -m "WIP: login form"
git stash push -u -m "WIP: include untracked files"
git stash list
git stash show -p stash@{0}
git stash pop
git stash apply stash@{0}
git stash drop stash@{0}

pop 会尝试恢复并删除 stash；apply 只恢复、不删除。重要改动优先使用 apply，确认无误后再 drop。

12. 标签与发布版本 ​

标签通常用于标记发布版本：

bash
git tag                       # 查看标签
git tag v1.0.0                # 创建轻量标签
git tag -a v1.0.0 -m "v1.0.0 release"  # 创建附注标签
git show v1.0.0               # 查看标签内容
git push origin v1.0.0        # 推送一个标签
git push origin --tags        # 推送所有本地标签
git tag -d v1.0.0             # 删除本地标签
git push origin --delete v1.0.0  # 删除远程标签

13. 冲突解决步骤 ​

发生冲突时，Git 会在文件中写入冲突标记：

text
<<<<<<< HEAD
当前分支的内容
=======
待合并分支的内容
>>>>>>> feature/login

处理步骤：

  1. 阅读上下文，决定保留、组合或重写代码。
  2. 删除 <<<<<<<、=======、>>>>>>> 标记。
  3. 保存文件并运行测试。
  4. 执行 git add 冲突文件。
  5. 合并时执行 git commit，变基时执行 git rebase --continue。

14. 推荐协作流程 ​

  1. 从最新的 main 或 develop 创建功能分支。
  2. 一项明确的改动对应一个或多个小而完整的提交。
  3. 推送功能分支并创建 Pull Request / Merge Request。
  4. 根据 Code Review 修改并补充测试。
  5. 通过 CI 后合并，删除远程功能分支。
  6. 本地执行 git fetch --prune，清理无效远程引用。

常见分支命名：

text
feature/user-login
fix/payment-timeout
docs/api-guide
refactor/order-service
hotfix/security-patch

15. 高频命令速查 ​

bash
# 状态与历史
git status
git diff
git log --oneline --graph --all

# 提交
git add .
git commit -m "feat: describe change"

# 分支
git switch -c feature/name
git switch main
git branch -d feature/name

# 同步
git fetch --prune
git pull --ff-only
git push

# 撤销
git restore 文件名
git restore --staged 文件名
git revert <提交ID>
git reflog

# 临时保存
git stash push -u -m "WIP"
git stash pop

16. 提醒 ​

  • 每次提交前执行 git status 和 git diff --staged，确认提交内容准确。
  • 不把密码、密钥、生产环境配置和大型构建文件提交到仓库。
  • 公共分支出现问题时优先用 git revert，避免使用会改写历史的强制推送。
  • 遇到误操作先执行 git reflog，大多数本地提交都能找回。
  • 对不确定的破坏性命令，先在测试仓库练习，或加上预览参数，例如 git clean -n。
